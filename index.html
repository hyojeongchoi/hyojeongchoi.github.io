04 / 09
Project 01 · Technical Deep Dive
핵심 구현 포인트
🏷️ ChatRoomRegistry 통합 설계 — OCP 충족
❌ 문제: StoryRoom과 CompanionRoom을 별도로 관리해 채팅방 타입이 늘수록 게이트웨이 로직이 복잡해졌습니다.
↓ 해결
✅ 채팅방 ID에 SR-/CR- 접두사 규칙을 도입하고 Registry 테이블로 통합. 신규 타입 추가 시 Registry 등록만으로 확장 가능하도록 설계했습니다.
✓ OCP 충족 · 확장 시 기존 코드 수정 없음
🔑 Redis SETNX + TTL — 원자적 초대코드
❌ 문제: 동행 채팅방 초대코드 중복 발급 방지와 만료 처리를 위해 별도 스케줄러를 두면 운영 복잡도가 증가합니다.
↓ 해결
✅ Redis SETNX로 원자적 생성 + TTL 10분 설정. 코드가 만료되면 Redis가 자동 삭제하여 스케줄러 없이 운영됩니다.
✓ 중복 발급 0건 · 스케줄러 없이 자동 만료
🔍 Projection + Pageable — 복합 필터 최적화
❌ 문제: 구장·날짜·인원 복합 필터 조회 시 전체 Entity를 SELECT해 불필요한 컬럼까지 로딩, 중복 쿼리도 발생했습니다.
↓ 해결
✅ JPA Close Projection으로 필요한 필드만 SELECT. @Query + Pageable로 페이징·정렬을 Repository 계층에서 처리해 계층 경계를 명확히 했습니다.
✓ 불필요 컬럼 제거 · DB I/O 최소화
📖 ChatReadCursor — 안읽음 메시지 추적
❌ 문제: 유저별 마지막 읽은 메시지 위치를 정확히 추적하지 않으면 안읽음 카운트 정합성이 깨집니다.
↓ 해결
✅ room_id + member_id UniqueConstraint로 유저별 커서를 보장. 조회 시 lastReadMessageId 이후 메시지 수를 쿼리해 정확한 안읽음 수를 반환합니다.
✓ 유저별 읽음 위치 정확 추적 · 정합성 보장
